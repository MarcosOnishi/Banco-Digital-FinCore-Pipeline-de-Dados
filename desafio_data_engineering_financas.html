01 — Contexto

Cenário do Desafio

O FinCore Bank é um banco digital com 2 milhões de clientes. O time de Dados precisa construir um pipeline robusto para processar transações financeiras, gerar indicadores de risco e disponibilizar dados analíticos para as áreas de Compliance, Risco de Crédito e Produtos de Investimento.

Você foi contratado como Engenheiro de Dados e sua missão é construir o pipeline de ponta a ponta, desde a ingestão dos dados brutos até as tabelas analíticas consumidas pelos cientistas de dados e analistas de negócio.

import pandas as pd import numpy as np import uuid from datetime import datetime, timedelta import random random.seed(42) np.random.seed(42) n = 100_000 # aumente para 500k em produção types_raw = ['PIX', 'pix', 'TED', 'ted', 'DOC', 'Boleto', 'BOLETO', 'Cartão', 'CARTAO', 'cartão'] categories = ['Alimentação', 'Transporte', 'Saúde', 'Educação', 'Entretenimento', 'Supermercado', 'Vestuário', 'Serviços', 'Viagem', None] base_date = datetime(2024, 1, 1) dates = [base_date + timedelta(days=np.random.randint(0, 365)) for _ in range(n)] # Introduz formatos mistos de data date_strs = [d.strftime('%d/%m/%Y %H:%M') if i % 3 == 0 else d.isoformat() for i, d in enumerate(dates)] amounts = np.round(np.random.lognormal(5, 1.5, n), 2) # Introduz valores negativos indevidos amounts[np.random.choice(n, 500, replace=False)] *= -1 ids = [str(uuid.uuid4()) for _ in range(n)] # Introduz duplicatas (~2%) dup_idx = np.random.choice(n, 2000, replace=False) for i in dup_idx: ids[i] = ids[np.random.randint(0, n)] df = pd.DataFrame({ 'transaction_id': ids, 'customer_id': np.random.choice(list(range(1, 10001)) + [None]*100, n), 'transaction_date': date_strs, 'amount': amounts, 'transaction_type': np.random.choice(types_raw, n), 'merchant_category': np.random.choice(categories, n), 'status': np.random.choice(['completed','failed','pending','reversed'], n, p=[0.85,0.07,0.05,0.03]), 'channel': np.random.choice(['app','web','atm','branch'], n, p=[0.6,0.25,0.1,0.05]), 'is_fraud_flag': np.random.choice([True,False,None], n, p=[0.02,0.83,0.15]), 'installments': [np.random.randint(2,13) if r < 0.3 else None for r in np.random.random(n)] }) df.to_csv('transactions.csv', index=False) print(f"Dataset gerado: {len(df)} linhas")

Ingestão e Camada Bronze

Ler os CSVs brutos e persistir na camada Bronze sem transformações.

O que fazer

  • Criar um script Python que leia os 3 CSVs e os persista em formato Parquet particionado por ano/mês (pasta bronze/transactions/year=2024/month=01/).

▸ Dica de implementação
import pandas as pd from datetime import datetime def ingest_to_bronze(filepath, output_dir): df = pd.read_csv(filepath) df['ingestion_timestamp'] = datetime.utcnow().isoformat() df['source_file'] = filepath # Particionar por ano/mês para transactions if 'transaction_date' in df.columns: df['year'] = pd.to_datetime(df['transaction_date'], errors='coerce').dt.year df.to_parquet(output_dir, partition_cols=['year','month'], index=False) return len(df)
02
Qualidade de Dados — Camada Silver

Tratar todos os problemas de qualidade e normalizar os dados.

Python SQL ETL PySpark
▾

Problemas a resolver em transactions

  • Duplicatas: Remover transações duplicadas por transaction_id, mantendo o registro mais recente de ingestão.
  • Datas: Padronizar todos os formatos de data para ISO 8601 (YYYY-MM-DD HH:MM:SS). Tratar datas inválidas como nulo.
  • transaction_type: Normalizar valores para uppercase sem acento: PIX, TED, DOC, BOLETO, CARTAO. Criar coluna transaction_type_clean.
  • Valores negativos: Transações com amount < 0 que não são do tipo "reversed" devem ser sinalizadas com data_quality_flag = 'NEGATIVE_AMOUNT'.
  • is_fraud_flag: Imputar nulos com False. Adicionar coluna fraud_flag_imputed = True quando imputado.
  • merchant_category nulos: Imputar com a string 'OUTROS'.

Métricas de qualidade esperadas

  • Gerar um relatório de qualidade com: % de duplicatas removidas, % de datas inválidas, % de nulos por coluna (antes e depois).
  • Salvar o relatório em logs/quality_report_YYYY-MM-DD.json.
▸ Exemplo de relatório de qualidade

03
Modelagem Analítica — Camada Gold

Criar tabelas analíticas agregadas para consumo pelos times de negócio.

SQL Python PySpark
▾

Tabelas a criar

  • gold.customer_transaction_summary — uma linha por cliente com: total de transações, volume total, ticket médio, tipo de transação mais frequente, data da última transação, total de transações com flag de fraude, e percentual de transações via app.
  • gold.monthly_volume_by_state — volume financeiro e quantidade de transações agrupados por mês e UF do cliente (requer JOIN com customers). Use window functions para calcular crescimento MoM (Month over Month).
  • gold.fraud_risk_profile — tabela de perfil de risco com: customer_id, credit_score, churn_risk_score, total_transactions, fraud_count, fraud_rate, avg_amount, investment_total, risk_segment (segmentação em "BAIXO", "MÉDIO", "ALTO" baseada nas métricas).
▸ Query SQL exemplo — Monthly MoM Growth
WITH monthly_agg AS ( SELECT DATE_TRUNC('month', t.transaction_date) AS month, c.state, COUNT(*) AS tx_count, SUM(t.amount) AS total_volume FROM silver.transactions t JOIN silver.customers c ON t.customer_id = c.customer_id WHERE t.status = 'completed' GROUP BY 1, 2 ) SELECT *, LAG(total_volume) OVER ( PARTITION BY state ORDER BY month ) AS prev_month_volume, ROUND( (total_volume - LAG(total_volume) OVER (PARTITION BY state ORDER BY month)) / NULLIF(LAG(total_volume) OVER (PARTITION BY state ORDER BY month), 0) * 100, 2 ) AS mom_growth_pct FROM monthly_agg;
04
Orquestração com Apache Airflow

Automatizar o pipeline com uma DAG que executa diariamente.

Airflow Python ETL
▾

Estrutura da DAG

  • Criar uma DAG chamada fincore_pipeline com schedule @daily.
  • Task 1 — ingest_raw: Executa o script de ingestão Bronze.
  • Task 2 — validate_quality: Roda as validações de qualidade e grava o relatório. Usa ShortCircuitOperator para abortar se a taxa de erro ultrapassar 5%.
  • Task 3 — transform_silver: Executa as transformações de limpeza.
  • Task 4 — build_gold: Cria as tabelas analíticas.
  • Task 5 — notify_success: Envia uma notificação (pode ser apenas um log com métricas do run).
▸ Estrutura da DAG
from airflow import DAG from airflow.operators.python import PythonOperator, ShortCircuitOperator from datetime import datetime, timedelta with DAG( dag_id='fincore_pipeline', schedule_interval='@daily', start_date=datetime(2024, 1, 1), catchup=False, default_args={'retries': 2, 'retry_delay': timedelta(minutes=5)} ) as dag: ingest = PythonOperator( task_id='ingest_raw', python_callable=ingest_bronze) validate = ShortCircuitOperator( task_id='validate_quality', python_callable=check_quality_threshold) silver = PythonOperator( task_id='transform_silver', python_callable=transform_to_silver) gold = PythonOperator( task_id='build_gold', python_callable=build_gold_tables) ingest >> validate >> silver >> gold
05
Detecção de Anomalias (Fraude)

Criar regras de negócio para identificar transações suspeitas.

SQL Python PySpark
▾

Regras a implementar

  • Regra 1 — Velocidade: Cliente com mais de 5 transações em 10 minutos deve ser flagrado como suspeito.
  • Regra 2 — Valor atípico: Transação com valor acima de 3 desvios padrão do histórico do cliente.
  • Regra 3 — Geo-impossível: Cliente realiza transação via branch em UF diferente da UF cadastrada no mesmo dia.
  • Regra 4 — Horário incomum: Transações entre 00h–05h com valor acima de R$ 5.000.

Crie uma tabela gold.fraud_alerts com todas as transações suspeitas, indicando qual regra foi acionada e um risk_score (número de regras violadas simultaneamente).

Desafio Bônus — Cloud & Streaming

Extensão do pipeline para ambiente cloud com dados em tempo real.

GCP / AWS Kafka Streaming
▾
⭐

Bônus: Esta tarefa é opcional e representa o nível Pleno/Sênior. Implemente ao menos um dos cenários abaixo.

  • Opção A — GCP: Reescrever o pipeline usando BigQuery como Data Warehouse, Cloud Storage para o Data Lake e Cloud Scheduler + Cloud Functions para orquestração. Usar bq load ou a API do BigQuery via Python.
  • Opção B — Streaming: Simular um produtor Kafka que emite novas transações a cada segundo. Criar um consumer que aplica as regras de fraude em tempo real e grava alertas em um tópico separado (fraud_alerts_stream).
  • Opção C — dbt: Substituir as transformações SQL das camadas Silver e Gold por modelos dbt, com testes de schema (not_null, unique, accepted_values) e documentação automática.
